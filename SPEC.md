---

## 1. Товч тодорхойлолт

Pi.mn нь гурван үүрэгтэй платформ:

1. **Танхимын сургалтын удирдлага** — анги, ирц, даалгавар, шалгалт, төлбөрийн хяналт
   (800 идэвхтэй танхимын сурагчийн ачааллыг даана)
2. **Онлайн сургалт + контент худалдаа** — бүлэг сэдэвчилсэн бодлого, тест, видео хичээлийг
   эрхийн (pass) системээр зардаг нийтэд нээлттэй дэлгүүр
3. **Адаптив ЭЕШ бэлтгэл** — сурагч бүрийн алдааны түүхээс суралцаж, яг хэрэгтэй
   бодлогуудыг нь сэдэв сэдвээр нь зааж өгдөг ухаалаг систем (§9 — гол давуу тал)

Бизнес загвар: **Freemium** — бүлэг сэдэв бүрийн эхний хэсэг (онол + эхний тест) үнэгүй,
үлдсэн нь төлбөртэй эрхээр нээгдэнэ.

| Зүйл | Шийдвэр |
|---|---|

| Backend API | **NestJS** (TypeScript) — EC2 дээр Docker-оор |
| Database | **PostgreSQL — AWS RDS** (автомат backup, өсөхөд Multi-AZ/replica) |
| Файл хадгалалт | **AWS S3** — зураг, бичлэг, PDF («Google Drive шиг») + **CloudFront CDN** |
| Сервер | **AWS EC2, Ubuntu Linux** — Nginx, Docker, GitHub Actions CI/CD |

| ORM | Prisma (type-safe, migration удирдлага) |
| AWS бүс | Сөүл (ap-northeast-2) / Сингапур — УБ-аас latency хэмжээд сонгоно |
| Төлбөр | QPay webhook → NestJS + дансны шилжүүлэг (гар баталгаажуулалт) |
| Хайлт | Глобал хайлт («Эндээс бүхнийг хайж олох…») |

Энэ стекийн үнэ цэнэ: vendor lock-in байхгүй, хүссэн үедээ өөр сервер/дата төв рүү
(Монголын дотоодын ч гэсэн) зөөж болно; auth, эрхийн систем, төлбөрийн логик бүгд
өөрийн кодод; улс даяарх хэмжээнд горизонталиар тэлэх зам нээлттэй (load balancer +
олон EC2). Анхаарах: сервер арчилгаа (security update, backup шалгалт, monitoring)
өөрсдийн хариуцлага болно — Docker + CI/CD-гээр аль болох автоматжуулна.

Frontend хостинг: эхний үед Next.js-ийг Vercel дээр (үнэгүй, глобал CDN, тохиргоогүй)
байршуулж, NestJS API-г AWS дээр ажиллуулахыг зөвлөж байна. Хүсвэл хожим frontend-ийг
ч AWS руу (CloudFront + EC2 SSR) бүрэн нүүлгэж болно — код өөрчлөгдөхгүй.

### 3.1 Гүйцэтгэлийн шаардлага

- **1000 хүн зэрэг орсон ч гацалтгүй** ажиллана. Үүний төлөө:
  - Нийтийн хуудсууд static + CDN edge cache — хэдэн зуун мянган хандалт ч даана
  - Динамик хэсэг: NestJS + RDS connection pooling (PgBouncer) + зөв индексүүд,
    Redis кэш (хожим, хэрэгцээ гарвал)
  - FB/Instagram шиг мэдрэмж: optimistic UI (товч дармагц шууд хариу үйлдэл),
    дараагийн хуудсыг урьдчилж ачаалах (prefetch), skeleton loading
- **800 идэвхтэй танхимын сурагч** + онлайн хэрэглэгчдийн өдөр тутмын ачааллыг даана
  (ирц, даалгавар, тест өдөр бүр бичигдэнэ)
- **10,000+ бодлого** орсон ч хурдан ажиллана: бүлэг сэдвээр хуваагдсан, индексжүүлсэн,
  хуудаслалттай (pagination)
- Бодлого бүр **мэдээлэл + зөв хариу + оруулсан огноо**-гоо найдвартай хадгална;
  бүх бичилт огноо/цагтай
- Бодлогын зургууд: thumbnail + CDN + lazy load — жагсаалт хуудас хүндрэхгүй
- Зорилт: хуудас нээгдэлт <1 секунд, дотоод үйлдлүүд агшин зуурын мэдрэмжтэй

## 6. Бүртгэл ба нэвтрэлт

### 6.1 Бүртгүүлэх хоёр зам
1. Хэрэглэгч **өөрөө** сайт дээр мэдээллээ өгч бүртгүүлнэ
2. Сурагч нэмэх эрхтэй **багш** шууд нэмнэ

### 6.2 Нууц үг
- Эхний нууц үг = **утасны дугаар**
- Анхны нэвтрэлтээр нууц үгээ солих санал гарна (заавал биш)

### 6.3 Танхимын сурагчаар бүртгүүлэх урсгал
1. Бүртгүүлэхдээ «Танхимын сурагч» сонговол **төлбөрийн хэсэг шууд нээгдэхгүй**
2. Багшаас **код** авч оруулсан үед л идэвхжинэ
   - Код = **тухайн өдрийн огноо** (ж: `20260611`) — автомат, багш юу ч үүсгэхгүй
   - Зорилго: сургалтын ачаалал, шинэ хүүхэд авах боломжийг багш өөрөө мэддэг тул
     дурын хүн өөрөө бүтэн жилийн ангид бүртгэгдэж орохгүй
3. Код зөв бол сурагч **«Ангид ороогүй сурагчид»** жагсаалтад овог нэр, профайлтайгаа орно
4. Багш тэндээс ангид нь оруулна → сурагчид **орсон өдрөөс хойшхи** мэдээлэл л харагдана
5. Анги солих/оруулах эрх нь багшийн интерфэйс дээр **ил, тод харагдана** (далд цэс биш —
   үндсэн ажлын хажууд байнга байх товч)

### 6.4 Онлайн сурагчаар бүртгүүлэх урсгал
- Код шаардлагагүй — төлбөр төлмөгц эрх **шууд автоматаар** нээгдэнэ
- Хоёр төрлийн эрх: **«Онлайн ангийн сарын эрх»** эсвэл **«ЭЕШ ангийн эрх»**
- Эрх нээгдмэгц бодлого, ном, бичлэгүүд нь харагдана
- Онлайн анги **багшгүй** — сурагч өөрийн хурдаар суралцана; ирц, даалгаврын хяналт,
  мийт/live хичээл байхгүй. Тест, бодлогын авто шалгалт + адаптив зөвлөмж (§9) л чиглүүлнэ

### 6.5 Анги сургуулийн сонголт
- Сурагч бүртгүүлэхдээ хэддүгээр ангийнх вэ гэдгээ (1–12) сонгоно
- 12-р ангийн хүүхдэд 12-р ангийн бүлэг сэдвийн бодлогууд харагдана

## 7. Контентын бүтэц: Ном → Бүлэг сэдэв → Бодлого

> ⚠️ Өмнөх «ном PDF-ээр бүтэн харагдана» гэснийг **хүчингүй болгов**.

- Ном **хэзээ ч бүтнээрээ нээгдэхгүй** (бүтэн PDF үзүүлбэл ашиггүй)
- Шатлал: **Ном → Бүлэг сэдэв → Бодлого**
- Бүлэг сэдвүүд нь **утасны тоглоомын үеүүд (levels) шиг** харагдана — эрхтэй бол нээгдэнэ
- Бодлогын token: `100-23-05` = 100x100 ном, 23-р хуудас, 5-р бодлого
- Бодлого бүр **оноотой**, дэлгүүрт зарагдах бол **үнэтэй** (Screen 4: дугаар/оноо/үнэ)

### 7.1 Бодлогын хариултын форматууд
1. **Тест** — сонголттой (A, B, C, D…)
2. **Тоо нөхөх** — ЭЕШ-ийн адил үсгийн оронд тоо бичиж нөхөх (a=…, b=…)
3. Шууд хариултын талбартай бодлого ч байж болно

Авто шалгалт: тест ба тоо нөхөх форматыг систем автоматаар шалгана.
Бодлогыг багш эсвэл админ нэмж оруулна.

### 7.2 Бодлогын гүн ангилал (таксономи) — адаптив алгоритмын суурь

> Хүнд/хөнгөн гэсэн 5 бүлэг маягийн бүдүүвч ангилал **хийхгүй** — бодлогыг бүтэц,
> бодох арга, асуултын хэлбэр, ашигладаг томьёогоор нь нарийн задалж ангилна.

Бодлого бүрийг оруулахдаа:

| Талбар | Жишээ |
|---|---|
| Бүлэг сэдэв | Тоо тоолол, Алгебр, Геометр… |
| Дэд сэдэв | Бутархай, Тэгшитгэл, Тойрог… |
| Бодох арга | орлуулах, хувиргах, зураг/дүрс ашиглах, тохиолдол шалгах… |
| Асуултын хэлбэр | утга олох, тэгшитгэл бодох, график унших, харьцуулах… |
| Ашигладаг томьёонууд | Виетийн теорем, синусын теорем, уламжлалын дүрэм… (томьёо бүр өөрөө бүртгэлтэй; нэг бодлого олон томьёотой холбогдоно) |
| Формат | тест / тоо нөхөх / хариулт |
| Оноо | тест дэх жин |
| Зөв хариу | системд хадгалагдана |
| Оруулсан огноо | автомат |
| Хүндийн зэрэг | **гараар оноохгүй** — датанаас автоматаар тооцогдоно (§9.3 Elo рейтинг) |

Нэг бодлого олон шошготой байж болно. Энэ таксономи нь статистик дүгнэлт болон
ирээдүйн ML-ийн feature-үүд болно — нарийн тэмдэглэх тусам систем ухаалаг болно.

Эдгээр шинж чанар нь §9-ийн алгоритмын түүхий эд болно — **бодлогыг сайн
тэмдэглэсэн байх тусам зөвлөмж оновчтой болно**.

### 7.2.1 Бодлого бүрийн нарийн шинжилгээ (`ProblemAnalysis`)

100x100/200x200 зэрэг эх сурвалжаас орж ирсэн бодлого бүр дээр эх текстийг өөрчлөхгүй,
тусдаа шинжилгээний давхарга хадгална. Энэ нь адаптив алгоритмын «амин сүнс»:

- **source trace:** аль PDF/DOCX, аль тест, A/B хувилбар, аль дугаараас орж ирснийг хадгална
- **answer key status:** хариу баталгаатай эсэх (`VERIFIED`), хариуны түлхүүр хоосон эсэх
  (`MISSING`), эсвэл хүн шалгах шаардлагатай эсэх (`REVIEW_REQUIRED`)
- **сэдэв/дэд сэдэв:** Тоо тоолол, Рациональ тэгшитгэл, Логарифм тэнцэтгэл биш гэх мэт
- **томьёо:** логарифмын тодорхойлох муж, модулийн тодорхойлолт, рациональ илэрхийллийн
  хуваарь тэг биш байх нөхцөл гэх мэт олон томьёо нэг бодлоготой холбогдоно
- **тодорхойлох муж:** хуваарь 0 болохгүй, язгуур доорх ≥ 0, log аргумент > 0,
  log суурь > 0 ба ≠ 1 гэх мэт
- **тэмдгийн дүрэм:** тэнцэтгэл бишийг сөрөг тоонд үржүүлэхэд тэмдэг эргэх, log/илтгэгчийн
  суурь 0 ба 1-ийн хооронд бол монотон буурах тул тэмдэг эргэх
- **алдах магадлалтай цэг:** төгсгөл нээлттэй/хаалттай андуурах, нэмэлт язгуур шалгахгүй
  орхих, хуваарийн тэгийг шийдэд оруулах, log-ийн тодорхойлох мужийг мартуулах гэх мэт

Автоматаар parse хийсэн metadata нь `AUTO_DRAFT`; хариу хоосон эсвэл source тодорхойгүй
бол `REVIEW_REQUIRED`; багш/админ шалгасны дараа `VERIFIED` болно. Ингэснээр систем
бодлогыг зүгээр нэг текст биш, математик бүтэцтэй дата гэж ойлгоно.

### 7.3 Бэлэн контент (эх сурвалж: `~/Desktop/future/`)

Контент аль хэдийн **сэдэв сэдвээр хуваагдсан** байгаа — оруулах ажил тэгээс эхлэхгүй:

- `100x100`, `200x200`, `300x300` (+ V2, ТОМ хувилбарууд) — сэдэв бүр тусдаа PDF/DOCX
  (ж: «Тригонометр-тэгшитгэл-1», «Интеграл-2», «Функц-3»…)
- `100x100 сэдвийн шалгалт`, `Сэдвийн шалгалтууд хуучин`, `Явцын шалгалт` — шалгалтууд
- `ESH-Sorilgo-2026`, `2024 ЭЕШ СОРИЛ` — ЭЕШ сорилууд
- `1000 бодлого`, `2025-1000 бодлого` — бодлогын сан
- `ПИ-ном` — П-тест-ном 1–11 (PDF + DOCX)
- `Logo` — √∞ лого (IMG_2438–2443)
- Бусад: Физик, 7,8-р анги, Хавтгай, ОГТОРГУЙ, Нөхөлт, ЭРГЭН САНАХ г.м.

Фолдерын бүтэц нь Ном → Бүлэг сэдэв шатлалтай шууд таарч байгаа тул **импортын
скриптээр хагас автоматаар** оруулна (файлын нэрнээс сэдэв танина).

`100x100 V2` импортын дүрэм:

- `future/100x100 V2` доторх 97 PDF-ийг A/B хувилбараар уншиж, `100V2` ном,
  сэдэв-тестийн chapter, онлайн тестүүд болгон upsert хийнэ
- `100x100-тестийн-хариу-A/B.docx`-оос answer key уншина
- хариу үсэгтэй бодлогууд `AUTO` шалгалтад орно; answer key хоосон бодлоготой тестүүд
  `MANUAL` болж, авто оноо бодохгүй
- бодлого бүрийн `ProblemAnalysis`-д томьёо, бодох арга, тодорхойлох муж, тэмдэг эргэх дүрэм,
  нийтлэг алдааг хадгална
- импортын audit report нь хэдэн PDF/тест/бодлого орсон, хэдэн хариу баталгаатай,
  хэдэн бодлого хүн шалгах шаардлагатайг JSON-оор гаргана
- classification report нь тест бүрийн бодлого бүрээр token, формат, question form,
  method, formula, domain note, sign rule, common mistake, review status-ийг бүрэн гаргана

## 8. Шалгалт / Тест модуль

- Танхимын болон онлайн сурагч **хоёуланд** ашиглагдана
- Тест **хугацаатай** (timer)
- **ЭЕШ стандарт формат: 36 сонгох + 4 нөхөх** — бэлэн загвар (template), багш нэг
  товчоор энэ форматын шалгалт үүсгэнэ. Нарийн бүтэц нь §8.1-д

### 8.1 36+4 форматын нарийн бүтэц (эх загвар: «1000 бодлого, Тест 18-A»)

- **Хугацаа:** 100 минут
- **I хэсэг «Сонгох даалгавар»:** 36 бодлого, тус бүр **5 сонголттой (A–E)**.
  Оноо хүнд рүүгээ өснө: №1–12 ≈ 1 оноо, №13–28 ≈ 2 оноо, №29–36 ≈ 3 оноо
- **II хэсэг «Нөхөх даалгавар»:** 4 бодлого. Бодолтын текст өгөгдөж, доторх
  `[a]`, `[b]`, `[de]`, `[fgh]` цонхнуудад цифр нөхнө:
  - Үсгийн тоо = орны тоо (`[de]` = 2 оронтой, `[fgh]` = 3 оронтой тоо)
  - Хасах тэмдэг бодолтод хэвлэгдсэн байж болно (`x = -[de]`)
  - Бодлого дотроо I, II, III, IV дэд хэсгүүдтэй, дэд хэсэг бүр өөрийн оноотой
    (1–7 оноо) — систем дэд хэсэг бүрээр оноо тооцно
- **Хувилбарууд:** нэг тест A/B хоёр хувилбартай (ангид хуулахаас сэргийлдэг) —
  систем тестийн variant бүлэглэлтийг дэмжинэ (Тест 18 = 18-A + 18-B)
- Цаасан хувилбарын дэвсгэр нь торон цаас — манай UI-ийн graph paper aesthetic-тэй
  яг таарна; онлайн хувилбар мөн адил харагдана
- Үүсгэх аргууд:
  - Шивж үүсгэх (Багш, Багш+, Админ)
  - **PDF-ээр оруулах** (Багш+, Админ)
- Тест/даалгавар оруулахдаа **аль ангиудад харагдахаа сонгоно** — автоматаар бүх ангид
  харагдахгүй
- **Тест бүрийн онолын хэсэг маш ойлгомжтой байна** — онол нь тесттэй хамт явна
- Дүн: тест формат автоматаар бодогдоно; дүнгүүд:
  - Сурагчийн «Шалгалтын дүн» хэсэгт (Screen 5)
  - **Багшийн самбарын «Шалгалтын дүн» хэсэгт** — ангиараа нэгтгэж харна
  - **Эцэг эхэд харагдана** — хүүхдийнх нь шалгалтын дүн бүрэн харагдана
- **Сэдвийн бүлгийн шалгалтын (36+4) дүнг багш оруулна** — өдөр тутмын өөрийн
  тэмдэглэгээнээс тусдаа, илүү найдвартай суваг
- **OMR импорт:** хариултын хуудас уншигч машины үр дүнг (CSV/Excel) системд
  оруулахад сурагч бүрийн бодлого бүрийн зөв/буруу автоматаар бүртгэгдэж,
  бүлэг сэдвийн сул талын дүгнэлт шууд гарна

## 9. Адаптив ЭЕШ бэлтгэлийн алгоритм ⭐ (гол давуу тал)

### 9.1 Гүйцэтгэлийн өөрийн тэмдэглэгээ (4 төлөв)

Сурагч бодлого бүр дээр **өөрөө** тэмдэглэнэ:

1. ✅ **100% ойлгож, алдалгүй бодсон**
2. 🔁 **Эхэлж бодоод алдсан → засаад зөв гаргасан**
3. ❌ **Алдсан / чадаагүй**
4. 🎯 **Буудсан** — таамгаар сонгоод оноо авсан

4 дэх төлөв маш чухал: сонгох тестэд хүүхдүүд алдаа гаргахгүйн тулд буудаж оноо
авдаг — үүнийг өөрөөр нь тэмдэглүүлснээр **«азын оноо»-г «жинхэнэ чадамж»-аас
ялгана**. Энэ бол хамгийн үнэ цэнэтэй датануудын нэг. Системийн авто үр дүн
(хариу зөв/буруу) давхар хадгалагдана — өөрийн тэмдэглэгээ + бодит үр дүн хоёул
дата болно.

### 9.1.1 Өдөр тутмын урсгал — утасгүй танхимын горим

Төв дээр ирмэгц сурагчдын утсыг хурааж, тарахад буцааж өгдөг тул **хичээл дээр
бодит цагт тэмдэглэх боломжгүй**. Тиймээс:

1. Сурагч **орой гэртээ хариад** утсаараа (эсвэл компьютерээр) нэвтэрч, өнөөдрийн
   бодсон бодлогуудаа 4 төлөвөөр **тэр оройдоо шууд** тэмдэглэнэ
2. **Шөнө бүр автомат ажил (cron):** анги бүрийн өдрийн дүгнэлтийг алгоритм бодож
   хадгална — хэн тэмдэглэсэн/тэмдэглээгүй, ангийн ахиц, сэдвийн зураглал
3. Багшид өглөө нь өчигдрийн ангийнхаа дүгнэлт бэлэн харагдана
4. Тэмдэглэгээний интерфэйс утас + компьютер хоёуланд төгс ажиллана (§2 responsive)

### 9.2 Урт хугацааны дата хуримтлал — 4 суваг

Оролдлого бүр бичигдэнэ: сурагч / бодлого / огноо цаг / авто үр дүн /
өөрийн төлөв (4 төлөвийн аль нь) / зарцуулсан хугацаа / **эх сурвалж**.

Эх сурвалж (суваг) нь:

1. **Өдөр тутмын өөрийн тэмдэглэгээ** — оройн утасны урсгал (§9.1.1)
2. **Онлайн тест** — системийн авто шалгалт
3. **Сэдвийн бүлгийн шалгалт** (36 сонгох + 4 задгай) — дүнг **багш** оруулна
4. **Хариултын хуудас уншигч машин (OMR)** — төвд удахгүй ирнэ: сканнердсан үр
   дүнг импортлоход сурагч бүрийн бодлого бүрийн зөв/буруу автоматаар орж,
   бүлэг сэдвийн сул талын дүгнэлт шууд гарна

Жил бүр асар их статистик хуримтлагдана — энэ нь хожмын machine learning-ийн
түүхий эд тул бүтцийг нь **эхнээс нь зөв зохиох** нь хамгийн чухал.

### 9.3 Тооцооллын арга — жинхэнэ математик аналитик, 3 үе шаттай

**1-р үе (эхнээсээ — ил тод статистик):**
- Шошго бүрээр (сэдэв, бодох арга, томьёо) амжилтын хувь — сүүлийн оролдлогод
  өндөр жин өгсөн weighted дундаж
- Бодлогын хүндийн **Elo маягийн рейтинг** — хэн бодоод чадсан/чадаагүйгээс хүнд
  нь датанаас өөрөө тодорно (гараар 1–5 гэж оноохгүй)
- Сурагчийн чадамжийн рейтинг — мөн адил датанаас

**2-р үе (хэдэн сарын дата):** магадлалын загвар — **IRT (Item Response Theory)**,
Khan Academy, Duolingo-гийн хэрэглэдэг арга: «энэ сурагч энэ бодлогыг зөв бодох
магадлал X%» гэж сурагч-бодлого хос бүрд тооцно.

**3-р үе (1 жилийн дата):** **machine learning — knowledge tracing**. Одооноос
датаг зөв бүтэцтэй цуглуулж байгаа нь яг үүний бэлтгэл; шаардлага гарвал ML-ийн
чиглэлээр гүнзгийрүүлж суралцана.

**Таамаглалын баталгаажуулалт (хамгийн чухал зарчим):** систем гаргасан таамаглал
бүрээ хадгалж, бодит шалгалтын дүнтэй харьцуулна — «алдана гэж таамагласан нь
үнэхээр алдсан уу» гэдгийг хэмжиж загвар тасралтгүй сайжирна. Үр дүнгийн жишээ:

> «Энэ хүүхэд *хувиргалттай*, *бутархайтай* бодлогон дээр алдах магадлал өндөр,
> *график унших* дээр сайн, одоогийн түвшингээр ЭЕШ-д **640–700 оноо** авах
> мужид явна»

### 9.4 ЭЕШ бэлтгэлийн зөвлөмж

- Профайл дээр үндэслэн **яг ямар бодлогуудаар бэлдэх хэрэгтэйг сэдэв сэдвээр нь**
  системээс шууд гаргаж өгнө
- Хүүхэд өөрийнхөө бодох хандлагыг (юун дээрээ алддагаа) **өөрөө мэдэж авна**
- Өмнө бодсон бодлогуудаа эргүүлж хараад **сэргээн санах** боломжтой — бие даан
  бэлдэхэд нь дөхөм
- Зөвлөмжид: алддаг сэдвийн бодлого + мартагдаж болзошгүй хуучин сэдвийн давталт
- **Таамаглах ЭЕШ оноо** — ойролцоо мужаар (ж: 640–700), дата нэмэгдэх тусам нарийсна
- **Өсөлт хөгжлийн мониторинг** — сар сараар чадамжийн өөрчлөлтийн график
- **Багшийн тооцоо:** ангийн зураглал (хэн юун дээр гацаж байгаа), тухайн анги нийт
  хэдэн бодлого бодсон, ямар тестүүд хийснийг автоматаар тоолж харуулна
- **Зан төлөв + чадамжийн судалгаа:** social media компаниуд хэрэглэгчийн үйлдэл
  бүрээс feed-ээ шийддэг шиг — сурагчийн зөв бодсон бүхнээс суралцаж, дараа юун
  дээр алдахыг урьдчилан хэлж, юу бэлдвэл амжилтад хүрэхийг зааж өгнө

### 9.5 Сул сурагчийн горим (хавар — шалгалт дөхөх үе)

100-аас 10-аад оноо авч буй сул сурагчид хэт олон, хэцүү бодлого өгвөл дарамт болно.
Тиймээс алгоритм сурагчийн түвшин + шалгалт хүртэлх хугацаанаас хамаарч **стратегиа
өөрчилнө**:

- Сул сурагчид: **амархан оноо зулгааж чадахуйц**, өөрийнх нь ойлгож чадах,
  **мэддэг гэхдээ алддаг** бодлогуудыг зааж өгнө
- Зорилго: цөөн төрлийн бодлогыг **алдахгүй болтол** сайжруулах — өргөн биш, гүн
- Сайн сурагчид: сул сэдвүүд рүү + илүү хүнд бодлого руу түлхэнэ

### 9.6 AI-ийн үүрэг ба «AI-г сургах» арга

«AI-г сургах» гэдгийг зөв ойлгох нь чухал — бид LLM-ийг өөрийг нь сургахгүй
(өртөг асар өндөр, шаардлагагүй). Оронд нь хоёр давхаргаар ажиллана:

1. **Сурдаг давхарга = §9.3-ын статистик/ML загварууд** — манай өөрийн дата дээр
   тасралтгүй сайжирч, тоон таамаглал гаргадаг хэсэг. Жинхэнэ «суралцаж байгаа» нь энэ
2. **Тайлбарлагч давхарга = Claude API** — загварын тоон дүгнэлтийг сурагчийн
   профайлтай хамт контекстээр өгөхөд хүний хэлээр зөвлөгөө, тайлбар бичнэ

Claude API-ийн хэрэглээнүүд (үе шаттай нэмнэ):

1. **Бодлогын туслах:** сурагч бодлого дээр гацвал AI-аас хинт (шууд хариу биш,
   чиглүүлэг) авна. AI-д бодлогын текст, зөв хариу, сэдвийн онол контекстээр өгөгдөнө
2. **Зөвлөмжийн тайлбар:** адаптив системийн дүгнэлтийг AI хүний хэлээр бичнэ
   («Чи тригонометр илэрхийлэл дээр сүүлийн 3 тестэд алдсан — эхлээд энэ онолыг
   давтаад, дараа нь эдгээр 5 бодлогыг бод»)
3. **Онолын асуулт-хариулт:** нээлттэй бүлгийн онолоос асуухад AI хариулна
4. (Хожим) Даалгаврын зургийг AI урьдчилж шалгаж багшид санал болгох

Анхаарах: API хэрэглээний төлбөртэй тул эрхгүй хэрэглэгчид хязгаартай
(ж: өдөрт 3 асуулт), эрхтэй сурагчид өндөр хязгаартай байна.

## 10. Даалгавар ба ирц

Төлөвүүд: **Хийгээгүй → Илгээсэн → Хийсэн** (эсвэл **Буцаасан** → дахин илгээнэ)

- **Онлайн зам:** сурагч дэвтрийн зураг/хариу илгээнэ → багш батална эсвэл буцаана
- **Танхимын зам:** багш ангид дэвтрийг биетээр шалгаад шууд «Хийсэн (ангид шалгасан)»
- Даалгавар **зурагтай** оруулж болно
- Харагдац: зөвхөн багш + сурагч өөрөө + эцэг эх. Ангийнхан хоорондоо харахгүй
- Ирц: багш өдөр бүр ангиараа бүртгэнэ

## 11. Эрхийн (named pass) систем

Нийтэд зарагдах бүх зүйл «эрх» гэсэн нэгдсэн ойлголтоор зохицуулагдана:

- Админ эрх үүсгэхдээ зааж өгнө:
  - **Нэр** (ж: «Сарын онлайн эрх», «ЭЕШ бэлтгэл багц», «Тоо тоолол бүлэг», «Бүх бодлого»)
  - **Хугацаа** (хэдэн өдөр/сар хүчинтэй)
  - **Хамрах хүрээ** (аль бүлэг сэдвүүд, ном, бичлэг, тестүүдэд хандах эрхтэй)
- Дурын хүн (худалдан авагч) бүлэг сэдвээр нь ч, бүгдийг нь ч худалдаж авч болно
- Хугацаа дуусмагц хандалт автоматаар хаагдана
- Онлайн сурагчийн «сарын эрх», «ЭЕШ ангийн эрх» ч мөн энэ системээр ажиллана

## 12. Төлбөрийн систем

### 12.1 Чөлөөт дүнгийн зарчим
- Төлбөрийн хуудсан дээр «N төгрөг төл» гэж **тулгахгүй**
- Оронд нь ангиудын тайлбар бүхий **мэдээллийн хэсэг** байна (аль анги юу вэ, үнийн заавар)
- Хэрэглэгч **төлөх дүнгээ өөрөө бичээд** тэр дүнгээрээ шууд төлнө

### 12.2 Төлбөрийн аргууд
- **QPay** — webhook-оор автоматаар баталгаажина
- **Дансны шилжүүлэг** — Багш+/Админ гараар баталгаажуулна
- (Бэлэн мөнгө — Багш+ гараар бүртгэнэ)

### 12.3 Төлбөр хяналтын алгоритм (маш сайн зохион байгуулалттай байх)
Төлбөрийн бичилт бүр:

| Талбар | Тайлбар |
|---|---|
| Сурагч/хэрэглэгч | Хэн төлсөн |
| Дүн | Өөрөө бичсэн дүн |
| Арга/зам | QPay / данс / бэлэн — төлсөн замыг бичнэ |
| Тайлбар | Чөлөөт текст |
| Төлсөн огноо, цаг | Яг хэзээ |
| Хамаарах эрх/сар | Аль сарын төлбөр, аль эрхэд |
| Төлөв | Хүлээгдэж буй / Баталгаажсан / Хугацаа хэтэрсэн |

Систем автоматаар:
- Сарын хугацаа дуусахын өмнө сануулна
- Хугацаа хэтэрсэн сурагчдын жагсаалтыг Багш+/Админд гаргана

## 17. Хөгжүүлэлтийн үе шат

**1-р шат — Суурь (MVP):**
Нэвтрэлт (утас=нууц үг, огнооны код), хэрэглэгчийн төрлүүд, анги удирдлага,
«Ангид ороогүй сурагчид» → хуваарилалт, ирц, даалгавар (2 зам), Багш/Багш+/Админ самбарын суурь

**2-р шат — Контент:**
Ном → бүлэг сэдэв → бодлого (тоглоомын түвшин UI), token + шинж чанарын метадата,
2 форматын бодлого + авто шалгалт, тест модуль (хугацаа, 36+4 загвар, PDF),
нийтийн preview хэсэг, видео хичээл, aesthetic нүүр хуудас

**3-р шат — Худалдаа:**
Эрхийн (pass) систем, QPay холболт, чөлөөт дүнгийн төлбөр, төлбөр хяналтын самбар,
хугацааны сануулга

**4-р шат — Адаптив алгоритм:**
3 төлөвийн тэмдэглэгээ + оролдлогын дата хуримтлал (энэ нь 2-р шатнаас эхэлж бичигдэнэ!),
алдааны профайл тооцоолол, ЭЕШ зөвлөмжийн хөдөлгүүр, сул сурагчийн горим,
«Миний сул талууд» хуудас

**5-р шат — Өргөтгөл + Апп:**
Эцэг эхийн самбар, өнгөний тэмдэглэгээ, тайлан. Дараа нь апп чиглэл — 2 алхамтай:
1. **PWA** — утсанд суулгаж болдог (home screen icon, push notification «багш даалгавар
   өглөө», offline кэш). Вэб кодоо шууд ашигладаг тул хурдан гарна
2. **Жинхэнэ нэйтив апп** (React Native/Expo) — backend, логик бүгд хуваалцагдана,
   зөвхөн UI давхарга шинээр. PWA хангалттай байвал заавал биш

> Чухал: оролдлогын дата цуглуулалтыг 2-р шатнаас л асаана — алгоритм 4-р шатанд
> бичигдэхэд дата аль хэдийн хуримтлагдсан байна.
